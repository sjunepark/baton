Drive GitHub agent work
without losing the wheel.

Baton turns your repo's issue/PR policy into deterministic commands — queue triage, safe worktree leases, CI & review inspection — so an AI agent can pick exactly one safe unit of work, do it in isolation, and report back.

📋 Facts, not guesses

Issues, PRs, checks, and review threads come back as stable JSON — no scraping the GitHub UI by hand.

🔒 Leased isolation

Edits only ever happen inside a leased worktree. Your primary checkout is never touched by automation.

🎯 One unit per run

baton next picks a single recommended action. No sprawling, multi-task agent runs.

01The mental model

Baton splits the job in two. Hold this picture and everything else follows.

⚙️ Baton CLI — the facts

Deterministic & replayable.

  • Reads GitHub issues, PRs, checks, review threads
  • Applies repo policy (labels, branch direction, keywords)
  • Picks one next action via baton next
  • Hands out & releases worktree leases
→

🧠 Codex + skill — the judgment

Where the thinking lives.

  • Decides how to implement the selected unit
  • Writes code, interprets ambiguous review feedback
  • Validates, pushes, comments, summarizes
  • Stops & escalates on human decisions

The CLI never writes your code; the agent never invents queue state. That separation is what makes runs safe to schedule unattended.

02The non-negotiable rules

Internalize these eight. They are the safety contract of every Baton run.

🧭Orient first. Run baton home or baton doctor when readiness is uncertain.
1️⃣Select before acting. Always baton next before unattended work, and handle exactly one selected unit.
🔑Lease before editing. baton lease … --json first. Work only inside the returned lease path.
🚫Never touch the primary checkout for automation work.
🤝Never merge unless the user explicitly asks.
📊Trust Baton output over manual GitHub browsing for issues, PRs, checks, threads.
🛑Stop & report on auth failures, lease conflicts, ambiguous scope, or human product/security/schema calls.
🧹Release clean. If release refuses a dirty worktree, report the path and changed files — don't force it.

05Branch: PR follow-up

When baton next returns pr-followup, you're babysitting an existing PR — not starting fresh.

🔍 Inspect in priority order

Required checks outrank review nits. Human threads outrank bots.

baton pr 142 --json
baton checks 142 --format toon
baton review-threads 142 --format toon

⬆️ Fix on the same branch

Lease the existing PR branch, fix failing required checks before review nits, validate, and push to that same branch.

Never open a new PR for follow-up work. A human unresolved, non-outdated thread may mean you stop and report instead.

🛑 Stop condition

Human unresolved review threads outrank bot comments and may require stopping entirely. Don't paper over a product/security/schema decision.

06Branch: issue intake

When baton next returns issue work, you're opening a fresh PR against the staging branch.

✅Confirm readiness. The issue must be implementation-ready with no skip labels.
🌿Branch from staging. Create work from the configured staging branch inside a lease: --base origin/agent --new-branch agent-work/<n>-slug.
🔗Reference, don't close. Open the work PR with Refs #<issue> — never closing keywords.
📤Closure is promotion's job. Work PRs don't close issues; the later promotion PR does.
Why the indirection?

Work PRs land on a staging branch so a human gates promotion to the default branch. Keeping closure on the promotion PR means an issue only closes when the change is actually shipped, not when an agent's draft merges to staging.

07--format toon vs --json

Same facts, two audiences. Pick by who's reading the output.

baton next --format toon

toon → agent context

Compact, low-token, human-skimmable. Use it for reads that feed an agent's context window: home, doctor, next, queue, checks, review-threads.

json → automation contract

Stable schemaVersion'd shape with the fields scripts depend on. Use it for anything a program parses, and for the result of mutating commands like lease/release/complete.

08Command index

Filter by name or category. Color tells you the risk class.

all read lease mutate setup
The mutation gate

Mutating commands require an explicit --apply, --yes, or a lease/release op. Do not infer permission to merge from any Baton output.

09Scheduling it as a Codex automation

Once the manual loop works, hand it to a recurring Codex app run — one safe unit per wake-up.

🧪 Prove it manually first

Never schedule implementation work until the same prompt succeeds once in a normal Codex thread — picking the right project, using Baton state, and respecting lease boundaries.

⏱️ Start slow, then tighten

PR monitor: every 15–30 min. Queue worker: 30–120 min. Digest: daily. Avoid overlapping runs — if a lease is active, report the conflict and stop.

Three prompt shapes

Queue worker

Select one safe issue/PR action, lease, work, validate, complete, release, stop.

PR follow-up

Babysit one PR until checks & review feedback are clear. Push to the same branch, never a new PR.

Read-only triage

Report the next action and exact future command. Never lease, edit, push, comment, or merge.

First-run review checklist

Before trusting the cadence, confirm each run: loaded the skill · ran baton next before acting · handled ≤1 unit · leased before edits · worked only in the lease path · stopped on human decisions · reported validation & release. Cleanup with baton leases / baton prune.
