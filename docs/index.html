Part 1

Concepts

What Baton is, and why it's shaped the way it is. Read this once and the commands explain themselves.

The problem Baton solves

Suppose small tasks are piling up across your projects: a bug here, a stale doc there, a review comment nobody answered. You'd like an AI coding agent to work through them on a schedule, unattended.

Try it, and three hard problems show up at once.

What should it work on? The agent needs a dependable list and a way to pick one item — without two scheduled runs grabbing the same task and colliding.

Where does it edit code? If it edits files in your everyday working copy, it can overwrite whatever you had open. Unattended edits need their own sandbox.

How do you keep it in bounds? Every project has rules: which branch to target, how to link an issue, what "done" means. A teammate learns these over time. An agent needs them written down and enforced on every run, or it drifts.

Baton answers these three questions the same way in every repository. It is a coordinator that sits between your GitHub project and the agent, turning a noisy repo into one clear, safe instruction.

The three-way split

The key realization is that "an AI agent does tasks on GitHub" is really three jobs — and the same thing should not do all three.

The first job is bookkeeping: reading GitHub, applying labels, working out which task is next, checking a PR against the rules. This must be exact and identical every time. Ordinary code is good at it; a language model is not.

The second job is judgment: making sense of a vague bug, deciding how to shape a fix, weighing a reviewer's comment. This is the creative part — the reason you wanted an agent at all.

The third job is isolation: providing a private directory where edits can happen without touching anything you care about.

Baton does the first job and refuses the other two. It computes facts and decisions. The agent — in practice Codex, guided by a small "skill" that teaches it Baton's rules — supplies the judgment. And whoever launches the agent supplies the sandbox.

Keeping the three apart is the whole design. Baton never writes code, because that's judgment. It never makes a sandbox, because that's isolation. It stays in its lane so the other two can trust it completely.

The name captures the flow. Picture a relay race, where the work is a baton: GitHub passes it to Baton, Baton passes a clear next step to the agent, the agent passes results back to GitHub. Nothing is dropped, and no one runs someone else's leg.

GitHub is the whole database

Baton keeps no operational database — no dashboard, no server to keep running. Your repository already holds the queue and policy state, and Baton just reads and writes it with discipline.

Your task list is your issues. You file work as a GitHub issue through a short structured form: what the problem is, what "done" looks like, how to check it. That issue is now a todo in the queue.

An issue's labels are a visible eligibility and lifecycle index. Baton uses a small, controlled set to record the kind of work, how much freedom the agent has, and how urgent it is. Because it only ever touches labels it owns, your own labels stay untouched.

A trusted, versioned issue comment bound to stable GitHub identity establishes Baton ownership. Ready and skip labels decide whether that managed issue is currently eligible for agent intake, but labels alone never enroll an ordinary issue.

A holding area between the agent and production

You don't want an agent's work landing straight on main. Even good work should be reviewed first. So Baton uses three layers of branch, and the middle one is the idea worth understanding.

main is production. Baton-managed batches reach it through deliberate reviewed promotions, while ordinary reviewed PRs may still merge directly and are later synchronized back into staging.

agent is a staging branch — a holding area. All Baton-managed work targets agent, never main. Think of it as an inbox where finished work collects and settles before you promote a batch to production.

agent-work/… branches are where a single task is built. The agent cuts one short-lived branch from agent, does one task, and opens a pull request back into agent.

Managed work flows inward: a reserved work branch merges into staging, and staging is later promoted to production. Ordinary repository development may still merge directly to main; Baton then recommends a reviewed merge-commit synchronization back into staging before more managed work.

This also explains a rule that surprises people: agent PRs say Refs #123, not Closes #123. A closing keyword would close the issue the instant its PR merged — but that PR only merges into staging. The task isn't truly finished until staging reaches production. So the work PR merely references the issue, and the issue closes later, at promotion. The alternative would mark tasks "done" while they're still a step away from main.

One task, chosen for you

When an agent wakes up, it shouldn't rummage through GitHub guessing. It asks Baton one question — what's next? — and gets the highest-priority candidate set plus the reason. Ties require an explicit choice before editing.

Baton chooses by a fixed order, because some work has to come before other work:

  1. Is delivery evidence incomplete? Baton reports a degraded observation and stops until the bounded ledger is repaired.
  2. Is staging unhealthy? Repair a failed staging branch or wait for its pending checks.
  3. Did base move directly? Synchronize base back into staging through a normal reviewed merge-commit PR.
  4. Does an open PR need another pass? Finishing work in flight beats starting something new; failing tests come before style nits.
  5. Is there a fresh issue to build? Only start something new when nothing in progress needs you.
  6. Is there something to investigate? Research-only issues are surfaced when a person asks.
  7. Nothing safe to do? Then Baton says so, and the agent stops.

Two smaller rules settle ties: a human's comment outranks a bot's, and a failing test outranks a cosmetic suggestion.

The important part: Baton only recommends. It returns a candidate and the reason; the agent picks one and acts. And an agent does exactly one task per run, then stops — so a scheduled run is always a single, reviewable step, never a runaway.

Rules that enforce themselves

The labeling and the PR checks aren't advice in a README. Baton runs them as GitHub Actions, so they hold even when no agent — and no person — is watching.

When an issue is opened or edited, Baton first writes its trusted ownership record, then sets controlled labels for work kind, autonomy, priority, and readiness. If a required section is missing, it leaves one updatable policy comment. Later body or label edits do not revoke ownership.

When a pull request is opened, Baton classifies intent from repository identity and branch shape. It validates only reserved work branches and the exact staging-to-base promotion pair; ordinary and fork PRs are unmanaged no-ops. Managed work must reference durably owned issues and use meaningful commits.

There's a subtle safety point worth calling out. A pull request can include edits to the policy code itself. If Baton ran the PR's copy of that code, a bad PR could switch off its own checks. So the check always runs the trusted copy from the base branch, loaded before the PR's code is present. Baton would rather refuse to run than run untrusted rules.

The boundary that never moves

Everything above serves one promise: an unattended agent never edits the code you're working on.

Baton keeps that promise by refusing to manage the sandbox at all. It will not create, reuse, or delete a working directory. That sounds like a gap, but it's deliberate. Whoever launches the agent already knows where it should run — a background checkout, a temporary worktree, a fresh clone. If Baton made its own, there'd be two competing owners of something the caller already controls.

So the contract is blunt:

Before an agent edits a single file, it must confirm it is in an isolated checkout that the caller prepared. If it can't be sure — if it might be in your everyday copy — it stops and reports that. It does not improvise a sandbox, and it does not ask Baton for one.

You now have the model: Baton does the exact bookkeeping, the agent does the judgment, the caller provides the sandbox, and GitHub holds the state. The Tutorial puts it to work.

Part 2

Tutorial

A guided walkthrough — install Baton, teach a repository its rules, file work, and run one safe task, with the reason for each step.

Getting Baton

Baton is a single command-line program written in Go, so installing it is one line. You also need the GitHub CLI, gh, signed in to your account — Baton discovers that credential for API calls, keeps it only in process memory, and never persists or prints it.

go install github.com/sjunepark/baton/cmd/baton@v0.7.0

If you're working on Baton itself, run it straight from source instead:

go run ./cmd/baton --help
go test ./...

From here on, baton means the installed command.

Seeing where you stand

Before changing anything, find out what Baton sees. Run it with no arguments, or use home:

baton home --format toon

This is always safe to run. Even in a repository Baton has never seen — no config, no auth, nothing — it won't error. It simply reports what's present and what's missing:

bin: ~/go/bin/baton
repo: sjunepark/baton
config: missing (.github/baton.yml)
auth: ok (gh auth token)
next: unavailable (config missing)
help[3]:
  Run `baton init --dry-run --json`
  Run `baton doctor --format toon`
  Run `baton --help`

Read that help section at the bottom. When something is missing, Baton usually names the exact command to fix it. That pattern repeats everywhere: Baton's output is built to hand you your next step.

For a real adoption gate rather than a glance, use doctor. It proves a live authenticated repository read, then checks exact managed files, workflows, Actions policy, rules, labels, ownership, delivery readiness, required checks, merge settings, and queues:

baton doctor --repo owner/name --format toon

A note on output formats

You'll see --format toon throughout. Baton's read-heavy commands present their output three ways, and you choose by who's reading.

Plain text (--format text, the default) is for you. TOON (--format toon) is a compact, stable layout meant to fit inside an AI agent's context. JSON (--json) is the exact, versioned contract for scripts. Setup and policy commands expose JSON directly with --json. The facts underneath are identical — you're just picking the shape that suits the reader.

Teaching a repository its rules

A fresh repository doesn't know Baton's conventions yet. Adopting it means writing a policy config, issue guidance and form, label list, and four GitHub Actions with explicit ownership and delivery boundaries.

Never do this blind. Every setup command previews first, so ask to see the plan before anything is written:

baton init --dry-run --json

Baton shows the eight files it would create:

.github/baton.yml                      # the policy config
.github/labels.yml                     # the labels Baton manages
.github/ISSUE_WORKFLOW.md              # how issues flow, for humans
.github/ISSUE_TEMPLATE/agent-work.yml  # the work-item form
.github/workflows/issue-policy.yml     # labels issues automatically
.github/workflows/pr-policy.yml        # checks PRs automatically
.github/workflows/work-item-transition.yml # completes merged work and promotions
.github/workflows/delivery-recorder.yml # records bounded delivery authority

Read that diff like any change to your repo. When you're satisfied, apply it — pinning the exact Baton version the Actions should install, so CI stays reproducible:

baton init --apply --go-install github.com/sjunepark/baton/cmd/baton@v0.7.0

Two pieces finish the setup. The staging branch has to exist, and the labels Baton relies on have to be present. Both preview by default and change nothing until --apply:

baton ensure-branch          # plan the `agent` staging branch
baton ensure-branch --apply

baton sync-labels --dry-run  # show which labels are missing
baton sync-labels --apply

Moving an older project that used a previous policy format? baton migrate-config --dry-run converts the old file into the new one, and --apply writes it. The theme is consistent: Baton shows, you approve, Baton applies.

For an existing repository, initialize and review the locked delivery ledger in shadow, then approve a separate change to sealed only after bootstrap comparisons pass. Finally run baton doctor --repo owner/name --format toon; adoption is not complete while it reports readyState: blocked.

Filing work as issues

In Baton, your todo list is your GitHub issues — so filing a task means creating a well-formed issue. Writing a clear one is judgment, which means it belongs to the agent skill, not the bare CLI. The skill command is $baton todo for a single item, or $baton todos to split a page of notes into several.

Underneath, it does two things: it checks how Baton would label the draft, then creates the issue.

baton issue-policy --body-file issue.md --json   # preview: which labels apply?
gh issue create --repo owner/name --title "..." --body-file issue.md

The decision that matters when filing is how much freedom to grant, and you should pick the least permissive level that fits:

  • Ready trivial — a tiny, obvious fix.
  • Ready bounded — scoped work with clear acceptance criteria.
  • Investigate only — research and report; no code changes.
  • Needs discussion — a person must decide before anyone builds it.

Write for the future, too. Describe the problem and what "fixed" looks like, not today's line numbers or function names — those will have moved by the time an agent picks the task up.

Looking at the queue

With a few issues filed, you can see what Baton would do without letting it do anything. Two read-only commands.

queue is the full picture: every open issue, which are eligible, which are held, and why.

baton queue --format toon --repo owner/name

next is the single recommendation — the one action Baton would take right now, and its reasoning:

baton next --format toon --repo owner/name

It names the candidate, the reason it won, and any instructions for handling it safely:

selectedAction: pr-followup
reason: failing-checks
candidates:
  - pullRequest #8  Fix branch policy  (headRef agent-work/branch-policy)
instructions:
  - Choose exactly one candidate.
  - Work in a caller-provided isolated checkout.
  - Push to the existing PR branch.

Research-only issues are held back from this recommendation, since they rarely want an automated code change. Ask for them directly when you want to inspect them:

baton next --action issue-investigation --format toon --repo owner/name

Doing one task

Here is the rule that matters most, so it stands alone:

Before editing any file, confirm you are in an isolated checkout that something else prepared. If you might be in your everyday working copy, stop. Baton will not make a sandbox for you.

With that settled, building an issue is a short sequence. Read snapshot, continue only when its outcome is actionable, and pick one candidate. Inside the isolated checkout, branch from the configured staging branch, make the smallest change that satisfies the acceptance criteria, validate it, and open a pull request back into that staging branch:

# In a caller-provided isolated checkout:
git switch -c <work_branch_prefix>123-short-slug <default_remote>/<staging_branch>
# implement, validate, push, and open a PR into the configured staging branch with "Refs #123"
# report the summary and validation evidence to the caller

Two ideas from Concepts show up here in practice: the PR targets the configured staging branch rather than the promotion branch, and it says Refs #123 rather than Closes, so the issue closes later, when staging is promoted.

Following up on a PR

Often the top recommendation isn't a new issue but an existing PR that needs another pass — failing tests, or review comments. Baton gives you the full picture cheaply, before you check anything out:

baton pr 42 --json --repo owner/name
baton checks 42 --format toon --repo owner/name
baton review-threads 42 --format toon --repo owner/name

Work in priority order: make the tests pass before worrying about style nits. And read the review threads carefully — a comment from a person can mean "stop and let a human decide," which outranks anything a bot said.

The fix goes on the same branch as the PR; you're continuing that work, not replacing it. So check out its head branch, fix only what this follow-up needs, validate, and push:

# In a caller-provided isolated checkout:
git switch <head-ref>
# fix, validate, and push to the same branch
# report the summary and validation evidence to the caller

Reporting the outcome

Whether you built an issue or followed up a PR, return a short summary and concrete validation evidence to the caller. The invoking automation owns execution history. GitHub PR and issue state owns semantic work-item completion; Baton does not keep a second local completion ledger.

Handing it to a schedule

Everything so far you drove by hand. The point of Baton is to let a scheduler drive it instead — but only once you trust the setup.

Prove it manually first. From the target repository checkout, run the read-only checks and one real task yourself, and confirm the agent stayed in its sandbox and never touched main:

baton doctor --repo owner/name --format toon
baton ensure-branch --json
baton sync-labels --dry-run --repo owner/name --json
baton snapshot --format toon --repo owner/name

Then schedule one worker whose whole job is one sentence: ask baton snapshot, act only for outcome: actionable, take exactly one candidate, work in an isolated checkout, and never merge. Recommendations are not claims, so do not run independent unattended dispatchers against the same repository. If you're unsure, start read-only and widen its permission once you've watched it behave. The skill command for a single safe run is $baton run --repo owner/name.

Step back and the pattern is the same everywhere: Baton shows you facts and a recommended next step, you or an agent supply the judgment, a sandbox you provide is where edits land, and nothing reaches production without a person. For unattended work, run baton snapshot and stop unless the outcome is actionable. The Reference below has the exact detail for every command.

baton doctor readiness checks read-only

Read the selected live repository and return doctor v2 readiness. Exact managed files, active trusted workflows, Actions policy, source-bound required checks, branch rules, merge settings, queues, labels, ownership, and delivery evidence must be compatible. baton doctor --repo owner/name --json

baton delivery-record / delivery-bootstrap delivery ledger reviewed · explicit apply

Initialize and pin a locked repository ledger through the generated Delivery Recorder workflow with delivery.authority: shadow. Review historical facts, relationships, and promotion comparisons; change authority to sealed only after the complete bootstrap passes. Reusing initialization with an existing locator performs a reviewed rollover from a drained predecessor into a different locked issue.

workflow_dispatch: mode=bootstrap-initialize, ledger_issue=900, observed_at=...
workflow_dispatch: mode=record

The plan and apply jobs share one run and an environment approval gate. See delivery bootstrap.

baton pr-transition advance merged work and promotions Actions · explicit apply

Plan or idempotently add the awaiting-review index after a work merge, or explicitly complete a sealed promotion. Promotion completion closes still-open delivered issues, removes the index, appends base-integration evidence, and commits the cursor last. A committed duplicate does not re-close later human reopens.

baton pr-transition --event "$GITHUB_EVENT_PATH" --dry-run --json
baton pr-transition --event "$GITHUB_EVENT_PATH" --apply --json

Flags: --event <path>, exactly one of --dry-run / --apply, --repo, --config, --json.

Skill commands

The Codex-facing router. $baton <command> <arguments> — the command name supplies the workflow, and everything after it is the argument. Bare $baton is a read-only menu, never permission to mutate.

CommandBehavior
$batonReadiness, queue summary, and 2–3 next commands. Read-only.
$baton status [repo]Readiness and setup checks. Read-only.
$baton next [repo]Show the next candidate set without acting.
$baton queue [repo]Show eligible and skipped issues/PRs.
$baton todo <todo>Create one Baton-ready GitHub issue. No branch or PR.
$baton todos <notes>Split notes into Baton-ready issues. No implementation.
$baton investigate <issue>Investigate and comment. No file edits.
$baton implement <issue>In an isolated checkout, build one issue and open/update a staging PR.
$baton follow-up <pr>In an isolated checkout, fix checks or review on the existing branch.
$baton run [repo]Choose exactly one safe candidate, handle it, then stop.
$baton adopt [repo]Dry-run/read-only target-repo setup checks.
$baton update [repo]Update an existing adoption via a normal reviewed PR. No merge.
$baton automate [repo]Explain or prepare scheduled one-unit automation.

Flags shared across commands. Read commands accept three output formats; the facts underneath are identical.

Configuration

Repository policy lives in .github/baton.yml. It is the source of truth for labels, branches, form sections, and PR rules. Full schema in CONFIG_SPEC.md.

Shape (condensed)

version: 1

setup:
  baseline_baton_version: v0.7.0     # release the setup files were last reviewed against

repository:
  default_remote: origin              # git remote used by Baton commands
  base_branch: main                  # production
  staging_branch: agent              # agent PRs target this
  work_branch_prefix: agent-work/    # must end with /

issue_policy:
  policy_comment_marker: '<!-- baton-issue-policy:v1 -->'  # stable; updates in place
  form_sections:                     # form heading → section id
    summary: Summary
    context_evidence: Context / evidence
    acceptance_criteria: Acceptance criteria
    agent_mode: Agent mode

  agent_mode_labels:

    Ready bounded: agent:ready-bounded

  skip_labels:          [needs-info, needs:discussion, needs:review]
  awaiting_review_label: needs:review

    ready-trivial: [summary, context_evidence, acceptance_criteria]
    ready-bounded: [summary, context_evidence, acceptance_criteria]

Required fields

version · repository.base_branch · repository.staging_branch · repository.work_branch_prefix · issue_policy.form_sections · issue_policy.agent_mode_labels · issue_policy.implementation_labels · issue_policy.skip_labels · issue_policy.awaiting_review_label · issue_policy.required_sections. Everything else uses the defaults above.

Validation rules

  • Branches satisfy Git ref-name rules, the remote is a normalized non-option name, work_branch_prefix ends with /, and neither base nor staging starts with that reserved prefix.
  • Every required section ID must exist in form_sections.
  • Every implementation label must appear in an agent-mode mapping or be marked externally managed.
  • Controlled label groups must not duplicate labels across unrelated groups.
  • When priority_labels is present, form_sections.priority and controlled_label_groups.priority are required, and its order defines queue rank.
  • The policy comment marker must stay stable once deployed, or old comments can't be updated in place.
  • The optional delivery locator is all-or-nothing and pins repository, locked issue, and checkpoint identities; sealed is enabled only after reviewed bootstrap.

Loading order & legacy mapping

Baton resolves config in order: explicit --config <path> → .github/baton.yml → legacy .github/agent-issue-policy.yml → built-in defaults (only for baton init, never for enforcement). Policy enforcement fails when no target-repo config exists. baton migrate-config maps the legacy file field-by-field into the new model.

Policies

The two rule sets Baton runs as GitHub Actions, so they hold with no agent watching.

Issue policy

On issue open/edit: parse the form sections, then add/remove controlled labels (work kind, agent mode, priority) and post one updatable policy comment when a required section is missing. Applies only labels it owns plus that single comment.

PR policy

Baton classifies every pull request from branch shape and repository identity, then validates only managed work and promotion flows:

  • Branch direction — work PRs target the staging branch, not main.
  • Reference keyword — the body must reference its issue with Refs.
  • Forbidden closing keywords — Closes/Fixes/Resolves are rejected on work PRs, so issues aren't closed before promotion.
  • Linked-issue ownership — every referenced issue has a trusted ownership record or temporary legacy fingerprint.
  • Immutable work evidence — each managed work decision appends the exact PR revision, prose digest, durable issue set, decision, and policy schema for post-merge delivery.
  • Commit-subject quality — rejects noisy subjects like fix, wip, update.
  • Commit-listing cap — fails closed when GitHub pagination prevents complete validation.
  • Promotion provenance — seals included work from immutable delivery records at the exact base/head/cursor. Closing references are optional presentation and must exactly match the seal when present; explicit post-merge transition owns closure.

Ordinary and fork pull requests are unmanaged no-ops. A same-repository agent-work/… branch is explicit Baton intent: it must target staging, and missing repository identity fails as indeterminate. Implementation and skip labels still guide issue intake, but edits after a managed PR opens do not change its merge policy. The retired allow_direct_base_branch_prs and reject_all_trivial_multi_issue_prs fields are accepted but ignored during migration.

Trusted execution. pull_request_target runs check out the trusted base SHA before installing Baton, so a PR can never make policy run its own modified code.

Workflows

Each recipe maps a skill command to the equivalent CLI calls, with its safety boundary. Full detail in USER_FLOWS.md.

$baton todo · todos create work issues only

Turn a request or a page of notes into Baton-ready GitHub issues, using the work-item form and a policy preflight. Creates issues only — no branch, commit, or PR.

baton issue-policy --body-file issue.md --json
gh issue create --repo owner/name --title "..." --body-file issue.md

Stop when: the repository is ambiguous, auth is missing, or the work is too vague to file honestly.

$baton status · adopt readiness & setup read-only / dry-run

From the target repository checkout, check whether it is Baton-ready and preview any missing setup. Nothing is applied.

baton doctor --repo owner/name --format toon
baton init --dry-run --json
baton migrate-config --dry-run
baton sync-labels --dry-run --repo owner/name --json
baton ensure-branch --json

Stop when: doctor is blocked, the install diff is unreviewed, legacy config is ambiguous, or branch state needs a human decision.

$baton queue · next inspect the queue read-only

See what Baton would do without acting: the eligible/skipped list, and the single recommended candidate set with its reason.

baton queue --format toon --repo owner/name
baton next --format toon --repo owner/name

Stop when: no eligible work, a blocked queue, or missing GitHub access.

$baton investigate research one issue comment only

For issues scoped to research or a recommendation. Inspect, run focused diagnostics, and comment findings. No file edits unless the user explicitly changes scope.

gh issue view <number> --repo owner/name
gh issue comment <number> --repo owner/name --body-file findings.md

Stop when: the issue isn't investigation-scoped, or findings need a product/security/schema decision.

$baton implement · run build one issue isolated checkout

Build one ready issue in a caller-provided isolated checkout, validate, and open/update a PR to the staging branch.

# in a caller-provided isolated checkout:
git switch -c <work_branch_prefix>123-short-slug <default_remote>/<staging_branch>
# implement the smallest change, validate, push, open PR with "Refs #123"
# report summary and validation evidence to the caller

Stop when: the isolated checkout is missing, acceptance criteria are ambiguous, the tree is dirty for unrelated reasons, or a human decision is required.

$baton follow-up fix an open PR isolated checkout

Address failing checks or review feedback on an existing PR, pushing to the same branch. No replacement PR, no merge.

baton pr 42 --json --repo owner/name
baton checks 42 --format toon --repo owner/name
baton review-threads 42 --format toon --repo owner/name
# in a caller-provided isolated checkout:
git switch <head-ref>
# report summary and validation evidence to the caller

Stop when: feedback needs human judgment, checks are red for unrelated reasons, or the branch can't be checked out safely.

$baton automate schedule one-unit runs after a manual run

From the target repository checkout, prepare scheduled automation only after a manual run has succeeded and respected isolation. Start read-only when uncertain; scheduled implementation still handles one unit, works in an isolated checkout, and never merges.

baton home --format toon
baton doctor --repo owner/name --format toon
baton ensure-branch --json
baton sync-labels --dry-run --repo owner/name --json
baton snapshot --format toon --repo owner/name

Stop when: doctor is blocked, no manual run has succeeded, auth is missing, isolation is unavailable, or the repo isn't Baton-ready.

Design & invariants

The boundaries the whole system upholds. Rationale in ARCHITECTURE.md and REQUIREMENTS.md.

Goals

  • Reuse one issue/PR policy across repositories.
  • Let automations ask for one safe next candidate.
  • Keep state inspectable through GitHub itself.
  • Be useful to both humans and agents.

Non-goals

  • General-purpose CI/CD platform.
  • Web dashboard or multi-tenant service (v1).
  • Fully autonomous merge by default.
  • Worktree leasing, pooling, or pruning.
  • Code generation inside the CLI.

Invariants

  • Automation never mutates the user's primary checkout.
  • Baton never creates, deletes, prunes, or leases worktrees — the caller owns isolation.
  • One run handles at most one unit of work.
  • GitHub Issues are the operational queue; target-repo config is the policy source of truth.
  • The CLI returns facts and decisions; the agent performs code changes.
  • Every mutating command has a dry-run, a pure planner, or an explicit --apply/--yes gate.
  • Human review outranks bots; unresolved human threads and failing required checks block merge.
  • pull_request_target never relies on PR-modified policy code.
  • Branch targets express routing, not ownership; trusted issue records and sealed delivery records are authoritative.
  • delivery.authority: shadow permits migration writes only; reviewed sealed authority enables policy, transition, and recommendations without history fallback.
