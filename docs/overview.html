Hand routine GitHub work
to an agent — safely.

Baton lets a solo developer give scheduled coding agents real issue and PR work on their personal repos, without building a platform or risking the working tree. Its one defining bet: the CLI owns the facts, the agent owns the judgment — and neither does the other's job.

🧠 Two brains

A deterministic binary emits verifiable facts & policy. A model supplies code, ambiguity, and escalation. Auditable and flexible.

🗂️ No new datastore

GitHub Issues, labels, PRs, checks, and a shared agent branch are the state. Nothing hides in a sidecar DB or dashboard.

🔒 Safe by construction

One unit per run, edits only inside a leased worktree, dry-run gates, trusted-code-in-CI, no auto-merge. Safety lives in the architecture.

00The bet — and the non-goals

Calibrate scope first. Baton is deliberately small; most of its design falls out of what it refuses to be.

Baton is

  • +A local-first coordinator for one solo developer's GitHub repos.
  • +A thin layer over GitHub infrastructure you already trust.
  • +A fact engine (a testable Go binary) plus a judgment agent (Codex + a skill).
  • +One binary: go install github.com/sjunepark/baton/cmd/baton@v0.1.5.

Baton is not

  • ×A general-purpose CI/CD platform or multi-tenant service.
  • ×A web dashboard or a replacement for Issues / Projects.
  • ×A fully autonomous merge bot.
  • ×A code generator — the CLI never writes a line of your code.

The target user runs scheduled Codex automations across personal repos; the secondary user is the Codex instances those automations launch. Everything below follows from serving both safely.

🧠 Codex + skill

Where the thinking lives.
  • Decides how to implement the selected unit
  • Interprets ambiguous review feedback
  • Writes code, validates, summarizes
  • Stops & escalates on human product/security/schema calls
Never re-derives GitHub state by hand. Never invents policy.

The contract between them is a set of schema-versioned objects — every structured result carries a kind and a schemaVersion. Facts flow down to the agent; code, comments, and PRs flow back up. Because the facts come from a testable binary, unattended runs stay auditable; because a model handles the fuzzy parts, they stay flexible.

⚑ Subtle truths the code reveals

Any open PR wins over fresh issues — even a fully green PR pre-empts new issue work; the check state only swaps the reason string. The enum reserves a sixth value, digest, but RecommendNext never emits it. And despite the architecture doc, the classifier does not yet read review threads — selection keys only off branch health and PR check state.

What makes an issue "eligible"?

Eligibility is pure label policy, not a heuristic. Toggle labels to resolve one issue:

An implementation label routes to issue-implementation; a comment-only label routes to issue-investigation; any skip label forces ineligible; and an implementation issue with an active linked PR becomes ineligible — but an investigation issue is never disqualified by a linked PR.

Decisions are sorted & de-duplicated, carry schemaVersion + kind, and are reproducible: same input → same bytes.

Two halves, opposite philosophies

⚑ A common misconception

An incomplete or blocked issue does not fail CI. Issue policy always exits 0 and enforces by writing the agent:blocked label plus one updatable comment. Only PR policy fails the build (exit 1). Enforcement is the label, not the exit code.

The branch model is the backbone

A work PR targets the agent staging branch, links its issue with the reference keyword (Refs), and must not close it. A promotion PR goes agent → main and must close the promoted issues (Closes/Fixes/Resolves). The two are mutually exclusive by design — and every keyword, branch, and label name is a baton.yml default, not hardcoded.

A dedicated git worktree checkout for exactly one unit. Its lease metadata lives under ~/.baton — a sibling leases/<id>.json (status, owner pid+host, expiresAt) plus the leases.lock mutex.
the only writable zone

The git branch layer is a separate plan/apply pipeline: Inspect reads SHAs, a pure Compute turns them into ordered commands plus errors/warnings, and Apply runs them only if there are no errors. It publishes or fast-forwards the staging branch only when safe, and warns instead of force-resetting a divergent branch. Execution stays outside Baton — acquire returns a path; Codex does the work there. The native git-worktree backend is v1, with the record designed to accept a pluggable backend later.

07The installer and the CI trust boundary

baton init scaffolds policy-as-code into a target repo. The PR-policy workflow then runs in a privileged context — which is exactly why it must run trusted code. This is the system's most important safety design.

The $baton router: consent that never escalates

The word after $baton selects the whole workflow; the rest is the true argument. Side effects climb one controlled rung at a time — and merge is fenced off above the top.

⛔ merge — only on an explicit user request and repo policy. Never inferred from Baton output.
Adoption escalates the same way

observe (read-only) → dry-run setup → a single manual $baton run → only then a scheduled one-unit worker. Note: $baton todo / todos are skill-only workflows (baton issue-policy + gh issue create) — the Go binary has no such subcommands.
