# Updating adopters to Baton v0.7.0

Baton v0.7 is a breaking reset to an issue-only Task manager. It removes
repository setup, issue-policy automation, branch and pull-request policy,
delivery tracking, compatibility projections, and the old config surface.
Normal v0.7 commands require only a repository, GitHub credentials, and an
issue carrying `baton:managed`.

This guide supports direct decommissioning from the default v0.5.0, v0.5.1,
and v0.6.0 installation surfaces. Do not pass through another old release.
Mixed pins, modified files, older releases, or unknown installations require
manual attribution; never choose the nearest known profile.

## 1. Inventory without changing anything

Run these read-only commands with an explicit repository:

```bash
repo=OWNER/REPOSITORY
default_branch=$(gh api "repos/$repo" --jq .default_branch)
branch_path=${default_branch//\//%2F}
gh api --method GET "repos/$repo/git/trees/$branch_path" -f recursive=1 --jq \
  'if .truncated then error("tree response truncated; inventory subtrees before continuing")
   else [.tree[] | select(.path | startswith(".github/")) | {path,type,sha}] end'
gh api "repos/$repo/actions/workflows?per_page=100" --paginate
gh api "repos/$repo/rulesets?includes_parents=true" --paginate --jq '.[].id' |
  while read -r ruleset; do gh api "repos/$repo/rulesets/$ruleset"; done
gh api "repos/$repo/branches?per_page=100" --paginate \
  --jq '.[] | [.name, .protected] | @tsv' |
  while IFS="$(printf '\t')" read -r branch protected
  do
    encoded=${branch//\//%2F}
    gh api "repos/$repo/rules/branches/$encoded?per_page=100" --paginate
    if [ "$protected" = true ]; then
      gh api "repos/$repo/branches/$encoded/protection"
    fi
  done
gh api "repos/$repo/labels?per_page=100" --paginate
gh api "repos/$repo/environments?per_page=100" --paginate
gh api "repos/$repo/issues?state=all&labels=baton:managed&per_page=100" --paginate
gh api "repos/$repo/pulls?state=all&per_page=100" --paginate
```

A missing or forbidden protection, ruleset, Actions, or environment response
is an unknown—not proof of absence. Record every workflow `go install` pin and
compare default-branch file bytes with the exact manifests under
`testdata/migration/`. Exact bytes can prove an unmodified Baton file; a
marker, similar filename, or nearby version cannot.

Compute SHA-256 for every known managed path and compare the result with the
matching exact-release manifest. Missing paths remain part of the inventory:

```bash
set -o pipefail
for file in \
  .github/baton.yml \
  .github/labels.yml \
  .github/ISSUE_WORKFLOW.md \
  .github/ISSUE_TEMPLATE/agent-work.yml \
  .github/workflows/issue-policy.yml \
  .github/workflows/pr-policy.yml \
  .github/workflows/work-item-transition.yml \
  .github/workflows/delivery-recorder.yml
do
  printf '%s ' "$file"
  if ! encoded_content=$(gh api --method GET "repos/$repo/contents/$file" \
    -f ref="$default_branch" --jq '.content // empty')
  then
    printf '%s\n' 'missing-or-forbidden'
    continue
  fi
  if [ -z "$encoded_content" ]; then
    printf '%s\n' 'unknown (content unavailable)'
    continue
  fi
  if ! digest=$(printf '%s' "$encoded_content" | tr -d '\n' |
    base64 --decode | openssl dgst -sha256)
  then
    printf '%s\n' 'unknown (content decode failed)'
    continue
  fi
  printf '%s\n' "$digest"
done
```

Failed path reads are reported as `missing-or-forbidden`; inspect the API error
and record an absent path as missing or an inaccessible path as unknown. The
commands do not write GitHub or repository state.

Also inventory branches, open and closed issues and pull requests, comments,
checks, delivery-ledger issues, environments, and local completion artifacts.
They are preservation targets, not cleanup targets.

## 2. Remove retired required checks first

Before changing workflows, remove `Check PR policy` and every other retired
Baton-produced context from branch protection and rulesets. Treat this as a
named manual GitHub settings change: review the exact rule and context in the
UI or another project-approved settings workflow. Do not continue while the
producer of a required context is unknown.

Re-run the read-only protection and ruleset queries. They must show that no
ordinary pull request requires a retired Baton context.

## 3. Review repository file removals

In an explicitly selected non-primary checkout, prepare a normal reviewed
diff. Remove exact, unmodified Baton-owned copies of these retired resources
when present:

- `.github/workflows/pr-policy.yml`
- `.github/workflows/work-item-transition.yml`
- `.github/workflows/delivery-recorder.yml`
- the old issue-policy or ownership-policy workflow
- an exact default `.github/baton.yml`, label manifest, generated issue
  guidance, or issue template that the project no longer wants

Inspect the proposed change with `git diff -- .github` and send it through the
project's ordinary review. Preserve modified or unmatched files and record the
manual decision they need. Baton v0.7 installs no replacement workflow or
config. A project may separately adopt its own issue template; Baton does not
install, pin, or track it.

Do not delete branches, issues, pull requests, comments, labels, check history,
locked ledgers, environments, worktrees, or local artifacts. Preserve v0.5
issue bodies and v0.6 ownership comments; those comments are inert history.

## 4. Review enrollment

Existing issues with `baton:managed` are already v0.7 Tasks. Classification is
derived from their fixed mode, priority, and blocker labels; no comment or
body fingerprint is read.

For v0.5 adopters, discover potential issues without enrolling them:

```sh
for label in agent:ready-trivial agent:ready-bounded agent:investigate-only
do
  gh issue list --repo "$repo" --state all --limit 1000 --label "$label" \
    --search '-label:"baton:managed"' --json number,title,state,labels,url
done
```

Old eligibility labels are discovery evidence only. Review each issue and
choose a mode, priority, and blockers. For every exact issue number approved
by a reviewer with no blocker, preview direct classified enrollment first:

```sh
baton --repo "$repo" --json enroll ISSUE --mode bounded --priority p2 --dry-run
```

Only after that per-issue preview is approved, run the same command without
`--dry-run`. Enrollment never edits the issue body or removes project labels.
It is valid to approve no old issues.

When the issue needs `needs-info` or `needs:discussion`, keep it blocked at
every intermediate state. If it has no fixed mode label, first preview and
apply `baton enroll ISSUE --dry-run` / `baton enroll ISSUE` without a mode,
then preview and apply one `baton update ISSUE --mode MODE --priority PRIORITY
--add-blocker BLOCKER`. The initial enrollment is blocked by its missing mode;
the prefix-safe update adds the blocker before making classification complete.

If the issue already has a fixed mode label, enrollment without a mode would
publish it as ready. Use the project's approved GitHub label workflow to add
the reviewed blocker before enrollment, verify the blocker is present, then
preview and apply classified `baton enroll ISSUE --mode MODE --priority
PRIORITY`. Leave the issue unenrolled if the blocker cannot be established
first.

## 5. Prove the final state

Repeat the inventory calls from step 1 and record unresolved actions. Success
requires all of the following:

- no retired Baton workflow is active;
- no retired Baton context is required by branch protection or a ruleset;
- every remaining modified, mixed, older, or unknown resource has a named
  owner and manual disposition;
- `baton --repo "$repo" list --state all` shows exactly the issues explicitly
  enrolled with `baton:managed`;
- v0.5 candidates not approved for enrollment remain ordinary issues.

The removed JSON contracts—including candidates, recommendations, snapshots,
operation reports, runs, PR/check/review data, and delivery records—have no
v0.7 equivalent. Project implementation and delivery tools remain outside
Baton's Task contract.
