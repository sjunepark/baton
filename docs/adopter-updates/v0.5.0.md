# Adopter Update: v0.5.0

This breaking pre-1.0 release removes `baton complete`, its flags, completion
JSON v1, and Baton's local `~/.baton/completions` ledger. The ledger was
write-only, used second-resolution identifiers, and duplicated execution state
already owned by callers such as Coda.

Existing completion JSON files are inert local artifacts. Baton does not
migrate or delete them; owners may archive or remove them independently.

## Migration

1. Remove `baton complete` from agent prompts, scripts, and `AGENTS.md` files.
2. Return work summaries, validation evidence, and remaining risk through the
   caller's normal Run or automation result channel.
3. Preview the new repository policy and transition workflow with
   `baton init --dry-run --json`.
4. After review, apply the managed files and run
   `baton sync-labels --dry-run --repo owner/name --json` before label apply.
5. Confirm `.github/workflows/work-item-transition.yml` uses the intended exact
   released Baton pin and the repository config's awaiting-review label.
6. Use direct GitHub comments only when repository policy explicitly requires
   a timeline comment; do not treat comments as execution state.

Maintained Coda code and tests do not invoke `baton complete`; Coda already owns
terminal Run state and needs no completion migration. Coda still consumes
`nextCandidates` v2 and `queueSnapshot` v1, so `baton next`, `baton queue`, and
their golden contracts remain supported during the documented snapshot-v1
adoption window.
