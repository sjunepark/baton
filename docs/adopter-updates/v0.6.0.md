# Adopter Update: v0.6.0

Promotion policy now seals the exact durable staged-work selection for a
promotion PR revision. Manual-only staging-to-base promotions pass without a
synthetic closing reference. Closing references are optional for every
promotion; if rendered, they must exactly match the sealed issue set. The
post-merge Work Item Transition workflow now closes delivered issues
explicitly, removes their awaiting-review index, records the resulting base
integration, and commits the promotion cursor last.

The sealed plan is bound to PR node/base/head, delivery cursor, coverage
watermark, included work records, and reviewed exclusions. Missing or partial
ownership/record/cursor/coverage/seal state fails closed. Failed GitHub requests
remain structured GitHub errors with exit code 5. There is no ancestry-history
or mutable merged-PR-body fallback after cutover.

This release also replaces target-first classification with explicit resource
ownership. Same-repository `agent-work/*` PRs are managed work only when they
target staging; another target is `misroutedWork`. Staging-to-base is
`promotion`. Candidate-shaped events with incomplete repository identity are
`indeterminate`. Ordinary PRs and known forks are `unmanaged` no-ops, whatever
branch they target.

Issue policy now writes a trusted `baton-managed-issue:v1` comment bound to the
issue node ID, plus a `baton:managed` index label. Queue intake, PR references,
and transitions use the ownership decision; labels alone no longer enroll an
issue. Existing matching issue-form bodies remain a temporary
`legacyFingerprint` source and are backfilled when issue policy next applies.
Body and label edits do not revoke a trusted record. Explicit unmanage is not
supported in this release.

Managed work-PR merge policy no longer reads current implementation, skip, or
trivial labels. Those labels still decide which issues Baton recommends and
starts, but an edit after a work PR opens does not silently change its required
check. Work PRs continue to require managed issue references, reject early
closing keywords, and enforce revision-bound commit rules. Transition apply
rechecks each referenced issue's durable ownership before the first write.

`pr_policy.reject_all_trivial_multi_issue_prs` is retired with
`allow_direct_base_branch_prs`. Existing v1 config may keep either field during
the migration window; Baton accepts and ignores them, and newly rendered config
omits them. Multi-issue work is permitted when every referenced issue is
managed. The changed config semantics and previously blocked PR outcomes are
part of the cumulative breaking `feat!:` v0.6.0 pre-1.0 minor classification.
`prPolicyDecision` is schema v4 and `workItemTransitionPlan` is schema v4 with
flows `work`, `promotion`, `misroutedWork`, `indeterminate`, and `unmanaged`.
Generated PR policy and transition workflows listen on all targets. PR Policy
conservatively admits every managed candidate shape; the transition workflow
admits promotions, while Delivery Recorder owns the record-before-work
transition path.

Transition apply preflights the exact merged PR revision, active seal, cursor,
bounded store, issue state, and durable ownership before writing. Partial issue
or record effects remain retryable in `operationReport`; the cursor checkpoint
is last. Once committed, a duplicate event is a total no-op and will not close
an issue that a human reopened later. Work Item Transition continues to need
only `contents: read`, `pull-requests: read`, and `issues: write`.

The same release adds and cuts over to the delivery ledger. `baton init`
installs `delivery-recorder.yml`, but recording is a successful no-op until the
optional `delivery` locator is reviewed into config. Bootstrap initialization
creates a canonical checkpoint in an existing locked issue; historical
bootstrap reports source facts, relationships, ambiguities, records, and
structured open-promotion shadow comparisons behind a stable plan ID. Every
mismatch blocks apply. Commit the initialized locator with `authority: shadow`,
complete and review bootstrap, then change it to `sealed` in a separate review.
Shadow is a migration gate, not a permanent dual-read mode.

Bootstrap now runs only through the generated Delivery Recorder workflow. Its
plan and apply jobs share a run ID and repository delivery concurrency group;
configure required reviewers on the `baton-delivery-bootstrap` environment.
Routine reconciliation also advances an exact coverage-only checkpoint after
a complete scan finds no missing managed record.
Generated PR Policy joins that concurrency group so promotion evaluation
cannot begin inside a delivery batch. Recorder rechecks are deduplicated and
deferred until the batch's final checkpoint/coverage commit. Both workflows
use GitHub's `queue: max` mode instead of the single-pending replacement
default.

Direct base development now yields `direct-base-work-pending` and a
`sync-staging` recommendation before PR or issue work. Maintainers open a normal
reviewed base-to-staging PR and use a merge commit; Baton records the exact
result but never opens, pushes, merges, squashes, rebases, or rewrites it.
Recorded merge/squash/rebase promotion results remain integrated even when the
result is absent from staging. Doctor blocks linear-history staging and
squash/rebase-only repository settings.

Public results are `pullRequest` v2, `prPolicyDecision` v4,
`workItemTransitionPlan` v4,
`deliveryRecordPlan` v2, `queueSnapshot` v2, `nextCandidates` v3,
`repositorySnapshot` v2, and `doctor` v2. This
remains the cumulative breaking `feat!:` pre-1.0 v0.6.0 minor; Release Please
retains ownership of generated release artifacts.

## Release Classification Audit

The release is breaking across every public SemVer surface Baton documents:
retired config fields change meaning and the delivery locator is new; versioned
JSON/TOON results changed; blocked doctor compatibility now exits `6`; and the
generated workflows changed triggers, permissions, concurrency, trusted
checkout, and installed files. These changes remain one cumulative `feat!:`
for pre-1.0 v0.6.0. Do not edit the manifest, changelog, tag, GitHub release, or
marked installer pins manually; Release Please owns them.

The two retired PR-policy fields remain decode-only for the v0.6 adopter
window, are never rendered, and should be removed from adopter config now.
Their decoder compatibility is scheduled for removal with the next config
schema major. Shadow delivery authority is a bounded migration gate, not a
permanent fallback or dual-read compatibility mode.

The `pullRequest` v2 dashboard now resolves merged managed-work references from
the covered staged-work record. Later PR title/body edits cannot change its
awaiting-review diagnostic; missing sealed evidence returns a warning instead
of treating mutable text as delivery state.

## Migration

1. Re-run `baton init --dry-run`, review the generated workflow diffs, then
   reconcile them so all-target triggers and split routing are installed: PR
   Policy admits every managed candidate shape, Work Item Transition admits
   promotions, and Delivery Recorder handles work and synchronization.
   PR Policy now needs `issues: write` to append and activate promotion seals,
   plus `checks: write` to invalidate a previously successful check if a second
   managed promotion opens.
2. Remove synthetic closing references from promotions when they are not useful
   to readers. If retained, make their issue set exactly match
   `promotionFacts.expectedIssues`; Baton no longer relies on GitHub auto-close.
3. Require the generated Work Item Transition workflow as the delivery
   completion authority and do not infer completion from PR prose alone.
4. Treat a missing policy result or incomplete delivery diagnostic as a
   verification failure. Repair ownership, staged records, checkpoint
   coverage, or the active seal with the dry-run planners, then retry.
5. After review, update `setup.baseline_baton_version` through the normal Baton
   adopter update workflow.
6. Re-run issue policy for existing form issues, or use the planned backfill
   tooling when available, until ownership diagnostics no longer report
   `legacyFingerprint`. Do not treat `baton:managed` alone as proof.
7. Remove `pr_policy.allow_direct_base_branch_prs` from repository config. Baton
   accepts but ignores it during this migration and no longer emits it.
8. Remove `pr_policy.reject_all_trivial_multi_issue_prs`. Baton accepts but
   ignores it during this migration and no longer emits it. Re-run managed work
   PR checks that were blocked only by implementation, skip, or all-trivial
   label state; the new decision uses durable ownership instead.
9. Before enabling the delivery locator, follow `docs/DELIVERY_BOOTSTRAP.md`,
   protect the `baton-delivery-bootstrap` environment, review the returned
   locator with `authority: shadow` and every shadow comparison, then manually
   reconcile once. Change the authority to `sealed` only after a complete
   reviewed plan. The
   new recorder requires `issues: write`, `checks: write`, `pull-requests: read`,
   and `contents: read`. PR Policy requires `issues: write` and `checks: write`.
   After the locator
   is committed, queue/snapshot/next and promotion checks fail closed on ledger
   gaps and never scan all closed staging PRs as fallback.
10. Run `baton doctor --repo owner/name --json` after the reviewed setup PR
    lands. Doctor v2 now reads the live repository; verifies the exact trusted
    files, workflow state, labels, ownership/delivery evidence, Actions
    allowlists, source-bound required checks, merge methods, and branch
    relationship; and
    blocks active merge queues. Organization hosted-runner policy remains an
    explicit degraded check when GitHub cannot expose it through supported
    REST. Do not finish adoption while `readyState` is `blocked`; follow each
    `checks[].remediation` and rerun the gate.
