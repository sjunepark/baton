# Adopter Update: v0.6.0

Promotion policy now derives the Baton work included between a promotion pull
request's base and head revisions. Manual-only staging-to-base promotions pass
without a synthetic closing reference. Promotions containing Baton work must
close every issue referenced by the included work PRs; an unrelated closing
reference no longer satisfies the policy.

Bounded commit comparisons or pull-request associations that cannot prove a
complete result fail closed through `prPolicyDecision.promotionFacts`, as does
an included work PR without the configured issue reference. The object contains
the expected issue set and a completeness flag as an additive schema-v1 field.
Failed GitHub requests remain structured GitHub errors with exit code 5 rather
than partially evaluated policy decisions.

Expected issues are read from the current title and body of merged work PRs.
Treat post-merge edits to that metadata as trusted policy changes.

## Migration

1. Update the trusted Baton install pin in `.github/workflows/pr-policy.yml`
   when this release is available. No additional workflow permissions are
   required beyond the existing `contents: read` and `pull-requests: read`.
2. Stop adding fake issue closures to manual-only promotions.
3. For issue-backed or mixed promotions, include configured closing references
   for every issue reported in `promotionFacts.expectedIssues`.
4. Treat `promotionFacts.complete: false` as a verification failure. Restore a
   missing work-PR issue reference, resolve GitHub availability, or reduce an
   oversized promotion as applicable, then retry.
5. After review, update `setup.baseline_baton_version` through the normal Baton
   adopter update workflow.
